โรงงานทำสมุด Ngoy Menghua เป็นหนึ่งในโรงพิมพ์ที่ใหญ่ที่สุดแห่งหนึ่งในประเทศไทย ธุรกิจอายุกว่า 70 ​​ปี ก่อตั้งเมื่อปี พ.ศ. 2494 เป็นผู้ผลิตสมุดที่มีชื่อเสียงโดยเฉพาะสำหรับโรงเรียนทั่วประเทศไทย ปัจจุบันธุรกิจครอบครัวนี้อยู่ภายใต้การบริหารของรุ่นที่ 3 แม้ว่าจะมีกระบวนการแบบเดิมอยู่มากมาย แต่ทีมผู้บริหารก็ได้ปรับปรุงระบบไอทีอย่างต่อเนื่อง และตั้งแต่ 10 ปีที่แล้ว Odoo ERP ก็ถูกนำมาใช้

Ecosoft เป็นบริษัทที่ปรึกษาด้านไอทีที่มุ่งเน้นโซลูชัน ERP แบบโอเพ่นซอร์ส 100% โดยมี Odoo นำเสนอมาอย่างยาวนานและ ERPNext เป็น ERP ตัวใหม่ล่าสุด

ในฐานะลูกค้ารายแรกของเรา Menghua และ Ecosoft มีความสัมพันธ์ที่พิเศษ เราเป็นพันธมิตรที่จะรับฟังข้อเสนอแนะซึ่งกันและกันเพื่อให้ได้สิ่งที่ดีที่สุด Ecosoft นำ ERP แรกไปใช้เมื่อ 10 ปีที่แล้วด้วย Odoo 7 ทีมผู้บริหารรู้สึกตื่นเต้นมากที่จะมีการดำเนินธุรกิจทางออนไลน์อย่างแท้จริง

ใช่แล้ว การใช้งานเป็นไปด้วยดีและให้บริการใบแจ้งหนี้เกือบ 40,000 ใบต่อปี ต่อไปนี้คือลักษณะของ Odoo 7 สำหรับ Menghua

![](~/assets/images/post/menghua-case-study_1.png)

ผู้ใช้ชอบมัน นักพัฒนาก็ชอบมัน Odoo เป็นเฟรมเวิร์กที่ทันสมัยและปรับแต่งได้ง่าย แล้วมีปัญหาอะไรถึงต้องเปลี่ยนระบบ?

ธุรกิจไม่เคยหยุดนิ่ง และระบบ ERP จำเป็นต้องพัฒนา หลายปีผ่านไปและไม่มีการอัปเกรดใดๆ ซอฟต์แวร์ที่ครั้งหนึ่งเคยทันสมัยก็ไม่ทันสมัยอีกต่อไป อีกไม่นานระบบจะต้องได้รับการอัปเดตที่สำคัญ แต่ด้วยระบบ Odoo เราดูแลมาเกือบสิบปีก็มีจุดที่ต้องคำนึงหลายประการ

## ความท้าทาย

ความท้าทายของ Menghua ไม่ได้เกี่ยวกับกระบวนการทางธุรกิจเพิ่มเติม เพราะในขั้นตอนสุดท้าย เรารู้ว่าเราสามารถเติมคุณลักษณะทางธุรกิจที่จำเป็นด้วย ERP ใดก็ตามที่เราเลือกได้ ความท้าทายมีมากขึ้นหลังจากเสร็จสิ้นการใช้งาน การบำรุงรักษาระยะยาว และการแก้ปัญหาในแต่ละวัน

ที่นี่เราจะพูดถึงจุดเสี่ยง 3 อันดับแรกที่อาจทำให้อายุของระบบไม่ยาวนานอย่างที่ต้องการ และเป็นเหตุผลที่ Menghua ตัดสินใจว่าจำเป็นต้องเปลี่ยนแปลง

### 1) ไม่สามารถอัพเกรดเป็นเวอร์ล่าสุดได้

เนื่องจาก Menghua ใช้ Odoo เวอร์ชั่น Community Edition จึงไม่มีทางอัปเกรดได้ง่ายๆ การอัปเกรดมีเฉพาะใน Odoo Enterprise Edition ซึ่งอาจมีค่าใช้จ่ายสูงเกินไป และยังมีอัพเดทเฉพาะกับ Odoo core เท่านั้น (ไม่ใช่โมดูลที่เราเขียนเอง)

แม้ว่า <a href="https://odoo-community.org">OCA</a> จะมีโปรเจ็กต์ชื่อ <a href="https://github.com/OCA/OpenUpgrade">OpenUpgrade</a> ที่สามารถใช้งานกับ Community Edition ปัญหาใหญ่กว่านั้นอยู่ที่สถาปัตยกรรมซอฟต์แวร์ของ Odoo เอง เพราะลักษณะการเขียนโมดูลของ Odoo นั้นมีลักษณะที่พันเกี่ยวกับส่วนอื่นๆมาก และเกือบจะเป็นไปไม่ได้ที่โมดูลที่คุณพัฒนาสำหรับเวอร์ชันเก่าจะเข้ากันได้กับ Odoo เวอร์ชันใหม่กว่า

OpenUpgrade เป็นการทำงานร่วมกันจากสมาชิกในชุมชนทั่วโลกเพื่ออัพเดทเฉพาะโมดูลของ OCA ข้ามเวอร์ชันต่างๆ แต่เมื่อพูดถึงโมดูลที่คุณทำเอง คุณก็สร้างอัพเดทเองอยู่ดี

สำหรับ Menghua เราได้ทำการเขียนโค้ดปรับแต่งระบบจำนวนมาก การอัปเกรด Odoo 7 เป็น Odoo 16 จึงเป็นไปได้ยาก (ทำใหม่จะง่ายกว่า)

### 2) การดูแลรักษาระบบในระยะยาวนั้นยุ่งยาก

ขอย้ำอีกครั้งว่า ผมไม่มีปัญหากับสภาพแวดล้อมการพัฒนาของ Odoo ที่จริงแล้วผมชอบมันมาก! ในฐานะโปรแกรมเมอร์ ผมยังไม่เห็นเฟรมเวิร์กอื่นใดที่ใกล้เคียงกับการพัฒนาโมดูลแบบกำหนดเองที่สามารถรวมเข้ากับกระบวนการทางธุรกิจที่มีอยู่ ติดตั้ง อัปเดต และถอนการติดตั้งได้อย่างน่าอัศจรรย์โดยไม่ต้องกังวลอย่าง Odoo

ปัญหาคือ ด้วยการเปลี่ยนแปลงฟีเจอร์เพียงเล็กน้อย จำเป็นต้องมีโปรแกรมเมอร์ที่เหมาะสมในการทำงาน

1. เซตอัพระบบแบบเดียวกับ production ที่ localhost
2. แก้ไขและทดสอบบน localhost
3. ส่งโค้ดไปที่ GitHub
4. SSH login ไปที่ server และ deploy

ในระหว่างโปรเจ็กต์หรือหลังจากใช้งานจริงไม่นาน การแก้ไขอย่างรวดเร็วไม่ใช่ปัญหาเนื่องจากทุกอย่างยังอยู่ในสมองของโปรแกรมเมอร์

แต่เมื่อหลายปีผ่านไป การแก้ไขข้อความเล็กน้อยในแบบฟอร์มที่พิมพ์ออกมาอาจต้องใช้เวลาหลายชั่วโมงในการทำ ตั้งแต่เซตอัพระบบถึงการ deploy

ลองนึกภาพว่าถ้าระบบ (OS, Libraries, Odoo ฯลฯ) ไม่ได้รับการอัปเดตเป็นเวลาอีก 10 ปี ก็จะเป็นไปไม่ได้เลยแม้แต่จะตั้งค่าเครื่องในเครื่องให้ทำงานในสภาพแวดล้อมแบบเดิมดังกล่าว

### 3) Odoo ที่มีการจัดการเต็มรูปแบบมี (managed cloud) มีค่าใช้จ่ายสูงเกินไป (EE เท่านั้น)

ประเด็นนี้ยังเกี่ยวกับการปรับปรุงโครงสร้างพื้นฐานให้ทันสมัยและความสะดวกในการบำรุงรักษาในระยะยาว ในอดีต โฮสติ้งคลาวด์ที่ได้รับการจัดการสำหรับ ERP ยังไม่พร้อม Menghua จึงไม่มีทางเลือกนอกจากต้องดูแล Hardware ทั้งหมดเอง โปรแกรมเมอร์เชื่อมต่อผ่าน SSH เมื่อต้องการปรับแต่งระบบ ปัจจุบันมีซอฟต์แวร์มากมาย รวมถึง <a href="https://www.odoo.sh">Odoo.sh</a> และ <a href="https://frappecloud.com/">Frappe Cloud</a> เป็นตัวเลือกที่ยอดเยี่ยมในการโฮสต์การผลิตของคุณอย่างปลอดภัยบนคลาวด์ พร้อมด้วยยูทิลิตี้ที่ช่วยให้ชีวิตของนักพัฒนาง่ายขึ้นในขณะที่ดูแลรักษาซอฟต์แวร์

สมมุติว่าเรายอมใช้ Odoo EE บน Odoo.sh ด้วยข้อมูลจำเพาะขั้นต่ำที่เป็นไปได้ (1 CPU, 25 GB) + สิทธิ์การใช้งานผู้ใช้ (ที่ผู้ใช้ 20 ราย) อาจทำให้ Menghua มีค่าใช้จ่ายถึง 21,652.52 บาทต่อเดือน หรือ 259,830.24 บาทต่อปี ซึ่งเป้นค่าใช้จ่ายที่มากกว่าการซื้อเซิร์ฟเวอร์สเป็คระดับไฮเอนด์ใหม่ทุกปีด้วยซ้ำ

## ERPNext ในฐานะตัวเลือกที่เหมาะสม

ต้องขอบคุณทั้ง Menghua ที่ให้ Ecosoft ช่วยปรับปรุงระบบอีกครั้ง และขอบคุณ ERPNext ที่มาถูกที่ถูกเวลา

จากประเด็นดังกล่าว Menghua และ Ecosoft ต้องตัดสินใจว่าจะปรับใช้ Odoo 16 CE + OCA อีกครั้ง หรือเปลี่ยนเป็น ERPNext ไปเลย จากการรวบรวมข้อดีและข้อเสีย สรุปได้ว่าความท้าทายข้างต้นทั้งหมดจะยังคงอยู่กับ Odoo CE ดังนั้นเราจึงตัดสินใจที่จะลองใช้ระบบ ERPNext แทน

ในส่วนถัดไป เราจะมาลงรายละเอียดเพิ่มเติมว่า ERPNext ว่าจะมาช่วยแก้ปัญหาต่างๆอย่างไร ก่อนอื่นผมขอแสดงรูปแบบใหม่ ERPNext ของ Menghua ให้ดูพอสังเขป

### 1) อัปเกรดได้ง่ายขึ้นแม้กับโมดูลที่สร้างเอง

Frappe Framework (ERPNext เป็นหนึ่งในโมดูลใหญ่) ไม่ได้เป็นโมดูลย่อยๆ (modular) เหมือนกับ Odoo ในทางกลับกัน มันเป็นแนวการสร้างแบบเนื้อเดียวกัน (monolith) มากกว่า ด้วยเหตุนี้ จึงไม่มีปัญหาเรื่องการพันเกี่ยวกันระหว่างโมดูลต่างๆแบบที่เราพบใน Odoo

การสร้างโมดูลและคุณลักษณะทางธุรกิจใหม่ๆ ที่รวมอยู่ในกระบวนการทางธุรกิจที่มีอยู่นั้นสามารถทำได้ผ่าน Event ต่างของ DocType ที่กำหนดไว้ล่วงหน้า (เช่น before insert, after insert, before save, after save ฯลฯ...) เมื่อเปรียบเทียบกับโอดู เทคนิคการเพิ่มโมดูลใหม่ๆใน ERPNext จะเป็นแบบหลวมๆมากกกว่า ทั้งนี้เพื่อแลกกับกระบวนการอัปเกรดที่ง่ายกว่า Odoo มาก โมดูลที่สร้างใหม่ของเรามักไม่ต้องการการเปลี่ยนแปลงใดๆ เพื่อให้เหมาะสมกับ ERPNext เวอร์ชันหลักถัดไป

ผมคงไม่สามารถลงรายละเอียดได้มากเกินไปที่นี่ แต่หวังว่าพอเห็นภาพ สำหรับ ERPNext การอัพเกรดระบบทำได้ง่ายเพียงแค่

1. Pull code
2. Run command > bench migrate

### 2) การบำรุงรักษาระยะยาวที่ง่ายขึ้นด้วยเครื่องมือแบบ Low Code

ก่อนหน้านี้ผมได้กล่าวไว้แล้วว่าการแก้ไขฉลากในแบบฟอร์มและปรับใช้กับการผลิตด้วย Odoo นั้นน่ารำคาญและใช้เวลานานเพียงใด ใน ERPNext หลังจากที่ระบบใช้งานได้แล้ว งานบำรุงรักษาส่วนใหญ่สามารถทำได้อย่างง่ายดายด้วยเครื่องมือที่ใช้ Low Code

และอีกมากมาย!

ตอนนี้ Menghua ใช้เครื่องมือเหล่านี้ ไม่เพียงแต่เพื่อแก้ไขสิ่งที่มีอยู่เท่านั้น แต่ยังปรับปรุงกระบวนการทางธุรกิจอื่นๆตามหน้างานจริงไปด้วย แน่นอนว่า มีหลายครั้งที่จำเป็นต้องมีการเขียนโค้ดที่ซับซ้อนกว่านี้ และจะต้องใช้วิธีการปรับใช้แบบคลาสสิก แต่ผมพบว่า 80% ของเวลาทั้งหมดสามารถทำได้โดยใช้เครื่องมือ Low Code ของ ERPNext

ใน Odoo ฟีเจอร์ Low Code เหล่านี้มีให้ใช้งานในแอป Studio (Odoo EE เท่านั้น) ซึ่งอย่างที่เราได้พิจรณากันแล้วว่ามีค่าใช้จ่ายที่สูงเกินไป

### 3) Frappe Cloud ที่มีการจัดการเต็มรูปแบบ

แล้วในที่สุด Menghua ก็สามารถใช้ประโยชน์จากแพลตฟอร์มซอฟต์แวร์ที่ทันสมัยและย้ายทุกอย่างไปยังระบบคลาวด์ได้ Frappe Cloud สามารถทำทุกอย่างที่ Odoo.sh ทำได้ โดยมีค่าใช้จ่ายเพียงเล็กน้อยเท่านั้น

ด้วยสเป็คเซิร์ฟเวอร์ที่คล้ายกัน แต่มาพร้อมกับผู้ใช้ไม่จำกัด จะมีค่าใช้จ่าย Menghua เพียง 892.55 บาทต่อเดือนหรือ 10,710.6 บาทต่อปี (Odoo.sh อยู่ที่ 259,830.24 บาทต่อปี)

![](~/assets/images/post/menghua-case-study_9.png)

ในอดีตปัญหาของเซิร์ฟเวอร์ต้องอาศัยเจ้าหน้าที่ไอทีในการจัดการ แต่ด้วย Frappe Cloud ทำให้ Menghua ERP หมดกังวล และด้วย support tickets แบบไม่จำกัดจากทีม Frappe Helpdesk เช่นกัน

![](~/assets/images/post/menghua-case-study_10.png)

และตอนนี้ อัปเดตเป็นเวอร์ชันล่าสุดสำหรับ Menghua ERP ได้ในคลิกเดียว หมดปัญหา!

## บทสรุป

ในขณะที่เขียนบทความนี้ Menghua ERP 2.0 ใหม่ที่เรียกว่า Menghua ERP 2.0 ใช้งานได้มาเกือบหนึ่งปีแล้ว ในตอนแรกเป็นการตัดสินใจที่ยากในการเปลี่ยนระบบ ERP โดยเฉพาะอย่างยิ่งเมื่อโซลูชันก่อนหน้านี้คือสิ่งที่เรา (Ecosoft) มีประสบการณ์มากกว่า ผมให้คำจำกัดความ ERPNext ว่าเป็นซอฟต์แวร์ที่สร้างขึ้นสำหรับผู้ใช้ มีหลายสิ่งที่ผู้ใช้สามารถทำได้เองโดยไม่ต้องอาศัยนักพัฒนา และสำหรับผู้ดำเนินการและผู้ดูแลในอนาคตเช่นเรา ก็ทำสิ่งต่างๆได้อย่างรวดเร็วและปวดหัวน้อยลง ทั้ง Frappe Cloud ก็ให้บริการเป็นอย่างดี มีปัญหาเกิดขึ้นบ้างระหว่างทาง แต่ปัญหาทั้งหมดได้รับการตอบกลับอย่างรวดเร็วโดยทีมงาน Frappe Helpdesk
