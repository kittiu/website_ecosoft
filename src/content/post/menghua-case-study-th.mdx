โรงงานโน๊ตบุ๊ค Ngoy Menghua เป็นหนึ่งในโรงงานโน๊ตบุ๊คที่ใหญ่ที่สุดในประเทศไทย ธุรกิจอายุ 70 ​​ปีขึ้นไป ก่อตั้งเมื่อปี พ.ศ. 2494 เป็นผู้ผลิตโน้ตบุ๊กที่มีชื่อเสียงโดยเฉพาะสำหรับโรงเรียนทั่วประเทศไทย ปัจจุบันธุรกิจครอบครัวนี้อยู่ภายใต้การบริหารของรุ่นที่ 3 แม้ว่าจะมีกระบวนการแบบเดิมอยู่มากมาย แต่ทีมผู้บริหารก็ได้ปรับปรุงระบบไอทีอย่างต่อเนื่อง และตั้งแต่ 10 ปีที่ผ่านมา Odoo ERP ก็ถูกนำมาใช้

Ecosoft เป็นบริษัทที่ปรึกษาด้านไอทีที่มุ่งเน้นโซลูชัน ERP แบบโอเพ่นซอร์ส 100% โดยมี Odoo นำเสนอมาอย่างยาวนาน และ ERPNext เป็นส่วนเสริมใหม่ล่าสุด

ในฐานะลูกค้ารายแรก Menghua และ Ecosoft มีความสัมพันธ์พิเศษกัน เราเป็นพันธมิตรที่จะรับฟังข้อเสนอแนะซึ่งกันและกันมากกว่า Ecosoft นำ ERP แรกไปใช้เมื่อ 10 ปีที่แล้วด้วย Odoo 7 ทีมผู้บริหารรู้สึกตื่นเต้นมากที่จะมีการดำเนินธุรกิจทางออนไลน์อย่างแท้จริง

ใช่แล้ว การใช้งานเป็นไปด้วยดีและให้บริการใบแจ้งหนี้เกือบ 40,000 ใบต่อปี ต่อไปนี้คือลักษณะของ Odoo 7 สำหรับ Menghua

![](~/assets/images/post/menghua-case-study_1.png)

ผู้ใช้ชอบมัน นักพัฒนาก็ชอบมัน Odoo เป็นเฟรมเวิร์กที่ทันสมัยและปรับแต่งได้ง่าย มีปัญหาอะไรเกี่ยวกับเรื่องนี้?

ธุรกิจไม่เคยหยุดนิ่ง และระบบ ERP จำเป็นต้องพัฒนา หลายปีผ่านไปและไม่มีการอัปเกรดใดๆ ซอฟต์แวร์ที่ครั้งหนึ่งเคยทันสมัยก็ไม่ทันสมัยอีกต่อไป อีกไม่นานระบบจะต้องได้รับการอัปเดตที่สำคัญ แต่ด้วยระบบที่มีอยู่ที่เราสนับสนุนมาหลายปี จึงมีความท้าทายร้ายแรงบางประการ

## ความท้าทาย

ความท้าทายของ Menghua ไม่ได้เกี่ยวกับกระบวนการทางธุรกิจเพิ่มเติม เพราะในขั้นตอนสุดท้าย เรารู้ว่าเราสามารถเติมคุณลักษณะทางธุรกิจที่จำเป็นด้วย ERP ใดก็ตามที่เราเลือกได้ ความท้าทายมีมากขึ้นหลังจากเสร็จสิ้นการใช้งาน การบำรุงรักษาระยะยาว และการแก้ปัญหาในแต่ละวัน

ที่นี่เราจะพูดถึงภัยคุกคาม 3 อันดับแรกที่ทำให้ซอฟต์แวร์ ERP มีอายุการใช้งานยาวนานดังที่เราพบ และเป็นเหตุผลที่ Menghua ตัดสินใจว่าจำเป็นต้องเปลี่ยนแปลง

### 1) ไม่สามารถอัพเกรดเป็นเวอร์ชั่นใหม่ได้

เนื่องจาก ERP ของ Menghua ใช้ Odoo Community Edition จึงไม่มีทางอัปเกรดได้ง่ายๆ การอัปเกรดมีเฉพาะใน Odoo Enterprise Edition ซึ่งอาจมีค่าใช้จ่ายสูงในการอนุญาตสิทธิ์ และยังมีผลกับ Odoo core เท่านั้น (ไม่ใช่การปรับแต่งของเรา)

แม้ว่า <a href="https://odoo-community.org">OCA</a> จะมีโปรเจ็กต์ชื่อ <a href="https://github.com/OCA/OpenUpgrade">OpenUpgrade</a> ที่ Rescue Community Edition ความท้าทายที่ยิ่งใหญ่กว่านั้นอยู่ที่สถาปัตยกรรมซอฟต์แวร์ของ Odoo สถาปัตยกรรมแบบโมดูลาร์ของเฟรมเวิร์ก Odoo นั้นเข้มงวดมากจนไม่น่าเป็นไปได้ที่โมดูลที่คุณพัฒนาสำหรับเวอร์ชันเก่าจะเข้ากันได้กับ Odoo เวอร์ชันใหม่กว่า

OpenUpgrade เป็นผลงานร่วมกันจากสมาชิกชุมชนทั่วโลกเพื่อย้ายเฉพาะโมดูลของ OCA ข้ามเวอร์ชันต่างๆ แต่เมื่อพูดถึงโมดูลที่คุณกำหนดเอง คุณยังคงอยู่ได้ด้วยตัวเอง

และด้วย Menghua ซึ่งเราได้ทำการปรับแต่งมากมายนอกเหนือจาก Odoo 7 การอัปเกรดเป็น Odoo 16 เป็นเพียงความฝันเท่านั้น

### 2) การดูแลรักษาระยะยาวนั้นเจ็บปวด

ขอย้ำอีกครั้งว่า ฉันไม่มีปัญหากับสภาพแวดล้อมการพัฒนาของ Odoo ที่จริงแล้วฉันรักมัน! ในฐานะโปรแกรมเมอร์ ฉันไม่เห็นเฟรมเวิร์กอื่นใดที่ใกล้เคียงกับการพัฒนาโมดูลแบบกำหนดเองที่สามารถรวมเข้ากับกระบวนการทางธุรกิจที่มีอยู่ ติดตั้ง อัปเดต และถอนการติดตั้งได้อย่างน่าอัศจรรย์โดยไม่ต้องกังวลเช่น Odoo

ปัญหาคือ ด้วยการเปลี่ยนแปลงฟีเจอร์เพียงเล็กน้อย จำเป็นต้องมีโปรแกรมเมอร์ที่เหมาะสมในการทำงานแบบคลาสสิก

1. ตั้งค่าสภาพแวดล้อมการพัฒนาภายในเครื่องที่จำลองสถานะการผลิต
2. เขียนโค้ดและทดสอบในเครื่องท้องถิ่น
3. กดโค้ดไปที่ GitHub
4. SSH เพื่อเข้าสู่เซิร์ฟเวอร์และปรับใช้

ในระหว่างโปรเจ็กต์หรือหลังจากใช้งานจริง การแก้ไขอย่างรวดเร็วไม่ใช่ปัญหาเนื่องจากทุกอย่างยังร้อนอยู่ในสมองของโปรแกรมเมอร์

แต่เมื่อหลายปีผ่านไป การแก้ไขฉลากขนาดเล็กในแบบฟอร์มที่พิมพ์ออกมาอาจต้องใช้เวลาหลายชั่วโมงในการตั้งค่าอินสแตนซ์ที่ทำงานอยู่ในเครื่องและชั่วโมงในการแก้ไขและปรับใช้

ลองนึกภาพว่าถ้าระบบ (OS, ไลบรารี, Odoo ฯลฯ) ไม่ได้รับการอัปเดตเป็นเวลาอีก 10 ปี ก็จะเป็นไปไม่ได้เลยแม้แต่จะตั้งค่าเครื่องในเครื่องให้ทำงานในสภาพแวดล้อมแบบเดิมดังกล่าว

### 3) Odoo ที่มีการจัดการเต็มรูปแบบมีค่าใช้จ่ายสูงเกินไป (และ EE เท่านั้น)

ประเด็นนี้ยังเกี่ยวกับการปรับปรุงโครงสร้างพื้นฐานให้ทันสมัยและความสะดวกในการบำรุงรักษาในระยะยาว ในอดีต โฮสติ้งคลาวด์ที่ได้รับการจัดการสำหรับ ERP ไม่พร้อมใช้งาน Menghua ไม่มีทางเลือกนอกจากต้องบำรุงรักษาโครงสร้างพื้นฐานฮาร์ดแวร์ในสถานที่ โปรแกรมเมอร์เชื่อมต่อผ่าน SSH เมื่อเขา/เธอต้องการปรับใช้แพตช์ ปัจจุบันมีซอฟต์แวร์มากมาย รวมถึง <a href="https://www.odoo.sh">Odoo.sh</a> และ <a href="https://frappecloud.com/">Frappe Cloud</a> เป็นตัวเลือกที่ยอดเยี่ยมในการโฮสต์การผลิตของคุณอย่างปลอดภัยบนคลาวด์ พร้อมด้วยยูทิลิตี้ที่ช่วยให้ชีวิตของนักพัฒนาง่ายขึ้นในขณะที่ดูแลรักษาซอฟต์แวร์

และแม้กระทั่งเราตัดสินใจยอมจำนนต่อ Odoo EE หากต้องการใช้ Odoo.sh ด้วยข้อมูลจำเพาะขั้นต่ำที่เป็นไปได้ (1 CPU, 25 GB) + สิทธิ์การใช้งานผู้ใช้ (ที่ผู้ใช้ 20 ราย) อาจทำให้ Menghua มีค่าใช้จ่าย 606.25 ดอลลาร์สหรัฐฯ ต่อเดือน หรือ 7,272 ดอลลาร์สหรัฐฯ ต่อปี ซึ่งมีค่าใช้จ่ายมากกว่าการซื้อเซิร์ฟเวอร์สเป็คระดับไฮเอนด์ใหม่ทุกสาขาทุกปี

## ERPNext มาช่วยแล้ว

เราขอขอบคุณ Menghua ที่ให้ Ecosoft ขับเคลื่อนเส้นทางแห่งการเปลี่ยนแปลง และขอขอบคุณ ERPNext ที่มาถูกที่และถูกเวลา

จากความท้าทายทั้งหมดที่กล่าวมา Menghua และ Ecosoft กำลังตัดสินใจว่าจะปรับใช้ Odoo 16 CE + OCA อีกครั้ง หรือเปลี่ยนเป็น ERPNext ทันที ด้วยการพูดคุยกันมากมายเกี่ยวกับข้อดีและข้อเสีย สรุปได้ว่าความท้าทายข้างต้นทั้งหมดจะยังคงอยู่กับ Odoo CE ดังนั้นเราจึงตัดสินใจลองใช้ ERPNext

จะหารือในรายละเอียดเพิ่มเติมว่า ERPNext เข้ามาช่วยเหลือได้อย่างไร นี่คือ ERP ใหม่สำหรับ Menghua ซึ่งจะมีลักษณะเช่นนี้

### 1) อัปเกรดได้ง่ายขึ้นแม้จะมีโมดูลที่กำหนดเองก็ตาม

Frappe Framework (ERPNext เป็นโมดูลขนาดใหญ่) ไม่ได้เป็นโมดูลาร์เหมือนกับ Odoo ในทางกลับกัน มันเป็นแนวทางแบบเสาหินมากกว่า ด้วยเหตุนี้ จึงไม่มีปัญหาเรื่องการขึ้นต่อกันระหว่างโมดูลต่างๆ ที่เราพบใน Odoo

การสร้างโมดูลและคุณลักษณะทางธุรกิจใหม่ๆ ที่รวมอยู่ในกระบวนการทางธุรกิจที่มีอยู่นั้นสามารถทำได้ผ่านเหตุการณ์ DocType ที่กำหนดไว้ล่วงหน้า (เช่น ก่อนแทรก หลังจากแทรก ก่อนบันทึก หลังจากบันทึก ฯลฯ...) โดยไม่ต้องกังวลมากนักเกี่ยวกับการขึ้นต่อกัน ทำให้การบูรณาการที่มีรายละเอียดน้อยลง เมื่อเปรียบเทียบกับโอดู เทคนิคการรวมระบบใน ERPNext แบบหลวมๆ นี้แลกกับกระบวนการอัปเกรดที่ง่ายกว่า Odoo มาก โมดูลที่ปรับแต่งเองของเรามักไม่ต้องการการเปลี่ยนแปลงใดๆ เพื่อให้เหมาะสมกับ ERPNext เวอร์ชันหลักถัดไป

ไม่สามารถลงรายละเอียดได้มากเกินไปที่นี่ แต่ฉันหวังว่าคุณจะเข้าใจบ้าง ใน ERPNext การโยกย้ายของทั้งระบบทำได้ง่ายเพียงแค่

1. ดึงรหัสล่าสุด
2. เรียกใช้คำสั่ง > การโยกย้ายแบบตั้งโต๊ะ

### 2) การบำรุงรักษาระยะยาวที่ง่ายขึ้นด้วยเครื่องมือที่ใช้โค้ดน้อย

ก่อนหน้านี้ฉันได้กล่าวไว้แล้วว่าการแก้ไขฉลากในแบบฟอร์มและปรับใช้กับการผลิตด้วย Odoo นั้นน่ารำคาญและใช้เวลานานเพียงใด ใน ERPNext หลังจากที่ระบบใช้งานได้แล้ว งานบำรุงรักษาส่วนใหญ่สามารถทำได้อย่างง่ายดายด้วยเครื่องมือที่ใช้โค้ดน้อย

และอีกมากมาย!

ใน Menghua เราได้รับพรจากฟีเจอร์ที่ใช้โค้ดน้อยเหล่านี้ ไม่เพียงแต่เพื่อแก้ไขสิ่งที่มีอยู่เท่านั้น แต่ยังปรับปรุงกระบวนการทางธุรกิจด้วย แน่นอนว่า มีหลายครั้งที่จำเป็นต้องมีการเขียนโค้ดที่ซับซ้อนกว่านี้ และจะต้องใช้วิธีการปรับใช้แบบคลาสสิก แต่ฉันพบว่า 80% ของเวลาทั้งหมดสามารถทำได้โดยใช้โค้ดเพียงเล็กน้อย ERPNext มาพร้อมกับคุณสมบัติเหล่านี้ตามค่าเริ่มต้น

ใน Odoo ฟีเจอร์โค้ดต่ำเหล่านี้มีให้ใช้งานในแอป Studio (Odoo EE เท่านั้น) ซึ่งถูกห้ามด้วยป้ายราคาที่สูง

### 3) Frappe Cloud ที่มีการจัดการเต็มรูปแบบ

ในที่สุด Menghua ก็สามารถใช้ประโยชน์จากแพลตฟอร์มซอฟต์แวร์ที่ทันสมัย และย้ายทุกอย่างไปยังระบบคลาวด์ได้แล้ว Frappe Cloud สามารถทำทุกอย่างที่ Odoo.sh ทำได้ แต่มีค่าใช้จ่ายเพียงเล็กน้อย

ที่นี่ด้วยสเป็คเซิร์ฟเวอร์ที่คล้ายกัน แต่มาพร้อมกับผู้ใช้ไม่จำกัด จะมีค่าใช้จ่าย Menghua เพียง $25 ต่อเดือนหรือ $300 ต่อปี (Odoo.sh อยู่ที่ 7,272 ดอลลาร์สหรัฐฯ ต่อปี)

## บทสรุป

ในขณะที่เขียนบทความนี้ Menghua ERP 2.0 ใหม่ที่เรียกว่า Menghua ERP 2.0 ใช้งานได้มาเกือบหนึ่งปีแล้ว ในตอนแรกเป็นการตัดสินใจที่ยากลำบากในการเปลี่ยนระบบ ERP โดยเฉพาะอย่างยิ่งเมื่อโซลูชันก่อนหน้านี้คือสิ่งที่เรา (Ecosoft) มีประสบการณ์มากขึ้น ฉันสรุป ERPNext ว่าเป็นซอฟต์แวร์ที่สร้างขึ้นสำหรับผู้ใช้ มีหลายสิ่งที่ผู้ใช้สามารถทำได้เองโดยไม่ต้องอาศัยนักพัฒนา และสำหรับผู้ดำเนินการและผู้ดูแลในอนาคตเช่นเรา สิ่งนี้ให้พลังในการทำสิ่งต่าง ๆ ได้อย่างรวดเร็วและมีความเครียดน้อยลง Frappe Cloud ก็ให้บริการเราเป็นอย่างดี มีปัญหาเกิดขึ้นบ้างระหว่างทาง แต่ปัญหาทั้งหมดได้รับการตอบกลับอย่างรวดเร็วโดยทีมงาน Frappe Helpdesk
